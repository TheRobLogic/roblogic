The big picture, four ways

There are a few words that get thrown around interchangeably in the tech world and make everything sound harder than it is. Let's untangle them. Pick your depth.

What's an "app?" What's a "repo?" What's a "package?"

Think of your Mac like a kitchen. An app is a finished appliance you plug in and use (like a blender). A package is the same thing, just delivered in a box through the side door instead of the front. A repo is the factory blueprint for building that blender from scratch.

You don't need to visit the factory. You just need the blender. Homebrew is the delivery service that brings it to your door and keeps it up to date.

An app is any program you run on your computer. Safari is an app. Excel is an app. Your Home Assistant dashboard is an app running on another computer.

A package is an app (or a piece of an app) that's been wrapped up for easy installation. When Homebrew installs something, it's downloading a package. When you went to install HACS on Home Assistant, it was asking you to install a package too, just from a different source.

A repo (short for "repository") is where the source code for a package lives online, usually on GitHub. It's the recipe, not the meal. You don't normally need to look at the repo unless you're curious how something works or you're troubleshooting a weird error.

The confusion happens because some installation instructions tell you to "clone the repo" when really they could just say "install the package." That's the developer showing their work instead of keeping it simple.

These three terms describe the same software at different stages of its life cycle:

  • Repository (repo): The source code, tracked in a version control system (usually Git, hosted on GitHub). This is where developers collaborate. It's like the master document in a shared drive.
  • Package: A built, versioned, distributable artifact created from that source code. Published to a registry (npm, PyPI, Homebrew, HACS). This is the shrink-wrapped product.
  • App: The running software on a device. What the end user interacts with.

The reason HACS (Home Assistant Community Store) can be painful: it blurs the line between "install a package" and "clone a repo," requiring users to interact with Git concepts they don't need to care about for other app stores.

At the technical level:

  • Repository: A Git-managed directory tree. Contains source, build configs, CI pipelines, docs. Hosted on a forge (GitHub, GitLab, Bitbucket). Every repo has a .git/ folder that stores the full commit history locally.
  • Package: An installable unit registered in a package manager. Homebrew formulae/casks, npm modules, Python wheels, Docker images. A package manager resolves dependencies, handles versioning (semver), and manages the install/uninstall lifecycle.
  • App: A user-facing process. Could be a native binary, an Electron wrapper, a web server (like Home Assistant), or a CLI tool. "App" is a UX term, not an architectural one.

Why HACS is annoying

HACS is a custom integration for Home Assistant that acts as a community package manager, but under the hood it's just cloning Git repos into your custom_components/ directory. It requires a GitHub account and a personal access token because it uses the GitHub API to download releases. This is architecturally lazy (they're offloading hosting to GitHub instead of running their own package registry), and it means you're debugging Git and API auth instead of just clicking "install." Homebrew doesn't have this problem because it runs its own registry.

What is Git?

Git is "Track Changes" for programmers. Like Track Changes in Word, but way more powerful. It remembers every version of every file you ever saved, lets you undo anything, and lets multiple people work on the same project without stepping on each other.

Git lives on your computer. GitHub is a website where people share their Git projects. They're related but different, like email (the technology) vs. Gmail (one service that runs it).

You probably won't use Git directly. But when installation instructions say "clone" something, they're asking you to download a Git project from GitHub to your computer. That's all.

Git is version control software that runs on your computer. It tracks changes to files over time, so you can always see what changed, when, and who did it. Every save point is called a "commit."

GitHub is a website (owned by Microsoft) where people post their Git projects so others can see, download, and contribute to them. Most open-source software lives on GitHub, including HACS, Home Assistant integrations, and Homebrew's package list.

Key terms you'll see around:

  • Clone: Download a Git project from GitHub to your computer
  • Pull: Get the latest updates from GitHub for a project you already cloned
  • Repo: Short for "repository," a single project tracked by Git
  • Branch: A separate copy of the project for trying things out without breaking the main version

When HACS asks you to create a GitHub personal access token, it's because it needs permission to download code from GitHub on your behalf. That's the part that trips people up. Homebrew skips all this because it handles the GitHub communication for you behind the scenes.

Git is a distributed version control system. Two important things to understand:

  • It's local-first. Every copy of a Git repo contains the entire history. You don't need internet access to use Git. This is by design: it was created by Linus Torvalds (who also created Linux) specifically so thousands of developers could work on the same codebase without a central server being a bottleneck.
  • GitHub is a hosting layer, not Git itself. GitHub adds a web interface, issue tracking, pull requests, and an API on top of Git. GitLab and Bitbucket are competitors that do the same thing. The code is the same regardless of where it's hosted.

For non-developers, the main interaction with Git is usually one of:

  • Installing software that asks you to "clone" a repo (download it)
  • Creating a GitHub account to get API access (like HACS requires)
  • Reading documentation that lives in a GitHub repo

The shift happening now: AI tools like Claude Code understand Git natively, so they can handle the Git workflow for you. You describe what you want, the AI manages branches, commits, and merges. The friction of learning Git commands is going away.

Git is a content-addressable filesystem with a VCS interface bolted on top. Key internals:

  • Object model: Every file, directory, and commit is stored as a SHA-1 (migrating to SHA-256) hashed object in .git/objects/. Three types: blobs (file contents), trees (directories), commits (snapshots with parent pointers).
  • Distributed: Every clone is a full replica. There's no "central" server, just conventions about which remote is canonical (usually origin).
  • Branching model: Branches are just pointers (40-byte text files) to commit hashes. Creating a branch is O(1). This is why Git branching is cheap compared to older VCS like SVN.
  • Staging area (index): Changes are staged before committing, giving you fine-grained control over what goes into each commit.

Homebrew actually uses Git under the hood: its formula repository (homebrew-core) is a Git repo. When you brew update, it does a git pull on that repo. But you never see or interact with that directly.

Why HACS is painful, architecturally

HACS clones Git repos directly into config/custom_components/. This means: (1) it needs a GitHub PAT for API rate limits, (2) updates are git pull operations that can fail silently if there are local modifications, (3) dependency resolution is nonexistent (no requirements.txt pinning across integrations), and (4) rollback requires manual git checkout. Compare to Homebrew, which pre-builds binaries ("bottles"), has proper dependency trees, and supports brew pin/brew switch for version management. HACS is a package manager built by people who were solving a community problem quickly, not architecting for long-term usability.

How Homebrew, Git, and Claude Code fit together

Homebrew is the hardware store. You walk in, ask for what you need, and walk out with it installed.

Git is the filing system behind the scenes that keeps everything organized and versioned. You don't need to think about it.

Claude Code is the knowledgeable friend who comes with you and says "you need this, let me set it up for you." You describe the problem, they handle the tools.

Here's how they work together in practice:

  • Homebrew installs software. You use it to put tools on your Mac, including Git and Node.js (which Claude Code needs to run).
  • Git tracks code changes. It's already on your Mac (Apple ships a version with the developer tools). Homebrew uses Git internally, but you don't interact with it directly unless you're doing software development.

The workflow: install Homebrew first, use it to install Node.js, then install Claude Code. After that, Claude Code can help you install and manage everything else.

Getting Started

Set it up in five minutes

Open Terminal (hit Cmd + Space, type "Terminal," hit Enter). Then paste these commands one at a time. Each one finishes before you run the next.

1

Install Homebrew

This one-liner downloads and installs everything. It'll ask for your Mac password (the cursor won't move while you type it, that's normal).

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

When it finishes, it may print some lines saying "Add Homebrew to your PATH." If so, copy and paste those lines too. If it doesn't say that, you're good.

2

Install Node.js

Claude Code needs this to run. One command, takes about 30 seconds.

brew install node
3

Install Claude Code

This installs it globally so you can run it from anywhere.

npm install -g @anthropic-ai/claude-code
4

Launch it

Type claude and hit Enter. A browser window opens for sign-in on first run.

claude
Account options: Claude Code works with a free claude.ai account (limited usage), a Pro subscription ($20/mo for heavier use), or pay-as-you-go API credits at console.anthropic.com (add $5, use until it runs out). Start free and upgrade if you like it.
