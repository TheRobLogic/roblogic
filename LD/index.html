The big picture, four ways

There are a few words that get thrown around interchangeably in the tech world and make everything sound harder than it is. Let's untangle them. Pick your depth.

What's an "app?" What's a "repo?" What's a "package?"

Think of your Mac like a kitchen. An app is a finished appliance you plug in and use (like a blender). A package is the same thing, just delivered in a box through the side door instead of the front. A repo is the factory blueprint for building that blender from scratch.

You don't need to visit the factory. You just need the blender. Homebrew is the delivery service that brings it to your door and keeps it up to date.

An app is any program you run on your computer. Safari is an app. Excel is an app. Your Home Assistant dashboard is an app running on another computer.

A package is an app (or a piece of an app) that's been wrapped up for easy installation. When Homebrew installs something, it's downloading a package. When you went to install HACS on Home Assistant, it was asking you to install a package too, just from a different source.

A repo (short for "repository") is where the source code for a package lives online, usually on GitHub. It's the recipe, not the meal. You don't normally need to look at the repo unless you're curious how something works or you're troubleshooting a weird error.

The confusion happens because some installation instructions tell you to "clone the repo" when really they could just say "install the package." That's the developer showing their work instead of keeping it simple.

These three terms describe the same software at different stages of its life cycle:

  • Repository (repo): The source code, tracked in a version control system (usually Git, hosted on GitHub). This is where developers collaborate. It's like the master document in a shared drive.
  • Package: A built, versioned, distributable artifact created from that source code. Published to a registry (npm, PyPI, Homebrew, HACS). This is the shrink-wrapped product.
  • App: The running software on a device. What the end user interacts with.

The reason HACS (Home Assistant Community Store) can be painful: it blurs the line between "install a package" and "clone a repo," requiring users to interact with Git concepts they don't need to care about for other app stores.

At the technical level:

  • Repository: A Git-managed directory tree. Contains source, build configs, CI pipelines, docs. Hosted on a forge (GitHub, GitLab, Bitbucket). Every repo has a .git/ folder that stores the full commit history locally.
  • Package: An installable unit registered in a package manager. Homebrew formulae/casks, npm modules, Python wheels, Docker images. A package manager resolves dependencies, handles versioning (semver), and manages the install/uninstall lifecycle.
  • App: A user-facing process. Could be a native binary, an Electron wrapper, a web server (like Home Assistant), or a CLI tool. "App" is a UX term, not an architectural one.

Why HACS is annoying

HACS is a custom integration for Home Assistant that acts as a community package manager. Under the hood, it downloads ZIP archives of releases from GitHub via the GitHub API. It requires a GitHub account and an OAuth authorization because it needs authenticated API access to avoid rate limits. This is architecturally lazy (they're offloading hosting to GitHub instead of running their own package registry), and it means you're debugging GitHub auth instead of just clicking "install." Homebrew doesn't have this problem because it runs its own registry.

What is Git?

Git is "Track Changes" for programmers. Like Track Changes in Word, but way more powerful. It remembers every version of every file you ever saved, lets you undo anything, and lets multiple people work on the same project without stepping on each other.

Git lives on your computer. GitHub is a website where people share their Git projects. They're related but different, like email (the technology) vs. Gmail (one service that runs it).

You probably won't use Git directly. But when installation instructions say "clone" something, they're asking you to download a Git project from GitHub to your computer. That's all.

Git is version control software that runs on your computer. It tracks changes to files over time, so you can always see what changed, when, and who did it. Every save point is called a "commit."

GitHub is a website (owned by Microsoft) where people post their Git projects so others can see, download, and contribute to them. Most open-source software lives on GitHub, including HACS, Home Assistant integrations, and Homebrew's package list.

Key terms you'll see around:

  • Clone: Download a Git project from GitHub to your computer
  • Pull: Get the latest updates from GitHub for a project you already cloned
  • Repo: Short for "repository," a single project tracked by Git
  • Branch: A separate copy of the project for trying things out without breaking the main version

When HACS asks you to sign in to GitHub (by entering a code on github.com/login/device), it's because it needs permission to download code from GitHub on your behalf. That's the part that trips people up. Homebrew skips all this because it handles the GitHub communication for you behind the scenes.

Git is a distributed version control system. Two important things to understand:

  • It's local-first. Every copy of a Git repo contains the entire history. You don't need internet access to use Git. This is by design: it was created by Linus Torvalds (who also created Linux) specifically so thousands of developers could work on the same codebase without a central server being a bottleneck.
  • GitHub is a hosting layer, not Git itself. GitHub adds a web interface, issue tracking, pull requests, and an API on top of Git. GitLab and Bitbucket are competitors that do the same thing. The code is the same regardless of where it's hosted.

For non-developers, the main interaction with Git is usually one of:

  • Installing software that asks you to "clone" a repo (download it)
  • Creating a GitHub account to get API access (like HACS requires)
  • Reading documentation that lives in a GitHub repo

The shift happening now: AI tools like Claude Code understand Git natively, so they can handle the Git workflow for you. You describe what you want, the AI manages branches, commits, and merges. The friction of learning Git commands is going away.

Git is a content-addressable filesystem with a VCS interface bolted on top. Key internals:

  • Object model: Every file, directory, and commit is stored as a SHA-1 (migrating to SHA-256) hashed object in .git/objects/. Three types: blobs (file contents), trees (directories), commits (snapshots with parent pointers).
  • Distributed: Every clone is a full replica. There's no "central" server, just conventions about which remote is canonical (usually origin).
  • Branching model: Branches are just pointers (40-byte text files) to commit hashes. Creating a branch is O(1). This is why Git branching is cheap compared to older VCS like SVN.
  • Staging area (index): Changes are staged before committing, giving you fine-grained control over what goes into each commit.

Why HACS is painful, architecturally

HACS downloads ZIP archives from GitHub releases via the REST API and extracts them into config/custom_components/. It authenticates via OAuth Device Flow (5,000 API calls/hour vs 60 unauthenticated). Architecturally: (1) no dependency resolution across integrations, (2) no rollback mechanism beyond manually replacing files, (3) a HA restart is required after every integration install. Compare to Homebrew, which pre-builds binaries ("bottles"), has proper dependency trees, and supports brew pin/brew switch for version management. HACS is a package manager built by people who were solving a community problem quickly, not architecting for long-term usability.

The Home Assistant Ecosystem

Making sense of Home Assistant's moving parts

Home Assistant is powerful, but its ecosystem has a lot of vocabulary that trips people up. "Integrations," "add-ons," "HACS," "custom components," "blueprints." Here's what each one actually means and why installing things can feel harder than it should.

Integrations, add-ons, and HACS: what's what?

Think of Home Assistant like a smartphone.

Integrations are built-in features, like how your phone already knows how to connect to WiFi and Bluetooth. Home Assistant ships with thousands of these. They let HA talk to your lights, thermostats, weather services, and more.

Add-ons are separate apps you install alongside HA. Like installing Spotify on your phone. They run independently but show up inside your HA interface. Things like a file editor, an MQTT broker, or Node-RED.

HACS is a community app store. Like sideloading apps on Android. It lets you install bonus integrations and pretty dashboard cards that the official team didn't build.

The confusing part: sometimes you need both an add-on AND an integration for the same thing. The add-on runs the service, the integration connects HA to it. It's like installing Spotify (the app) and then logging into it (the connection).

Integrations

Integrations are how Home Assistant connects to your devices and services. When you add a Philips Hue bridge, a weather service, or a Zigbee coordinator, you're setting up an integration. HA ships with over 2,800 built-in integrations. You set them up in Settings → Devices & Services → Add Integration. No files to download, no terminal commands.

Custom integrations are the same thing, but made by community developers instead of the HA team. They add support for devices or services that aren't built in yet. These are what HACS installs for you.

Add-ons (now called "Apps")

Add-ons are completely separate programs that run alongside Home Assistant. They each get their own isolated environment. Examples: a code editor for your config files, an MQTT message broker, a Zigbee bridge, or Node-RED for visual automations.

You install them from the built-in add-on store at Settings → Add-ons. One click to install, one click to start.

Important: Add-ons are only available if you're running Home Assistant OS (the full operating system install). If you installed HA as a Docker container, you don't have add-ons. You'd run those same services as separate Docker containers yourself. This is the single biggest source of "why can't I find the add-on store?" confusion.

HACS

HACS (Home Assistant Community Store) is a community-built integration that adds a second "store" to your HA. It lets you browse and install community-made integrations, custom dashboard cards (like nicer-looking buttons and graphs), and themes.

It requires a GitHub account because it downloads everything from GitHub and needs to authenticate to avoid rate limits. That's the part that gives people headaches. Once it's set up, it works like a regular store with browse, install, and update.

Integrations (core vs custom)

Core integrations ship in the HA source at homeassistant/components/<domain>/. Each is a Python package with a manifest.json, __init__.py, and platform files (sensor.py, light.py, etc.). They go through code review, CI, and are tested against each HA release.

Custom integrations go in <config>/custom_components/<domain>/. Structurally identical to core integrations. HA loads custom_components/ at startup, and a custom integration with the same domain name overrides the core version. Custom integrations must include a version key in manifest.json.

Add-ons (the Supervisor layer)

Add-ons are Docker containers managed by the Supervisor, which is itself a Docker container that orchestrates others via the Docker API. The Supervisor handles image pulls, container lifecycle, networking (bridge network 172.30.32.0/23), and an ingress reverse proxy that lets add-on web UIs appear inside the HA frontend without exposing extra ports.

Each add-on has a config.yaml defining its Docker image, required ports, volumes, environment variables, and AppArmor profile. Add-on repos are Git repositories with a directory per add-on.

Why add-ons require HA OS/Supervised: The Supervisor is the orchestrator. Without it, there's nothing to pull images, manage containers, or provide ingress routing. HA Container users must run equivalent services as standalone Docker containers and connect them via integrations manually.

HACS internals

HACS is a custom integration installed to custom_components/hacs/. It does not use git clone. It calls the GitHub REST API to fetch repository metadata, then downloads ZIP archives of tagged releases (or default branches) via https://github.com/{owner}/{repo}/archive/refs/tags/{tag}.zip. It extracts the relevant subtree (custom_components/<domain>/ for integrations, dist/ for frontend cards) and places files in the correct config directories.

Authentication uses GitHub's OAuth Device Flow (not a PAT). The user authorizes at github.com/login/device, granting read-only public repo access. Authenticated requests get 5,000 API calls/hour vs 60 unauthenticated. HACS checks all tracked repos for updates on startup, so users with many repos can still hit rate limits during peak load.

Blueprints and automations

An automation is a rule: "when X happens, do Y." Like: "when motion is detected, turn on the light."

A blueprint is a template for a rule. Someone else wrote the automation, left some blanks for you to fill in (which motion sensor? which light?), and shared it. You fill in the blanks and you've got a working automation without writing anything from scratch.

Blueprints live on the HA community forums. When you find one you like, you import it with a URL. No GitHub account needed, no HACS, no terminal.

Automations are the bread and butter of Home Assistant. They follow a trigger-condition-action pattern:

  • Trigger: Something happens (motion detected, sun sets, temperature drops below 65)
  • Condition: Optional filter (only if it's after 6pm, only if someone is home)
  • Action: Do something (turn on lights, send a notification, lock the door)

You can build automations in the visual editor (Settings → Automations) or write them in YAML.

Blueprints are reusable automation templates that community members share. Instead of figuring out the YAML yourself, you import a blueprint and fill in the blanks. Examples: "motion-activated lights with timeout," "notify when a door is left open," "set thermostat based on time of day."

To import: Settings → Automations → Blueprints → Import Blueprint. Paste a URL from the HA forums. Done.

Blueprints are the easiest way to get complex automations running without learning YAML or asking anyone for help.

Blueprints are the lowest-friction way to get value from the HA community. No accounts, no tokens, no restarts. Import a URL, pick your devices, done.

Blueprints are YAML files with !input tags that define user-configurable placeholders. They live on disk at:

# Blueprint directory structure
<config>/blueprints/
    automation/
        homeassistant/           # Built-in blueprints
            motion_light.yaml
        <author>/               # Imported blueprints
            my_blueprint.yaml
    script/
        <author>/
            script_blueprint.yaml

A blueprint defines a blueprint: block with metadata (name, description, domain, input definitions) followed by the automation/script body that references those inputs via !input input_name. When you create an automation from a blueprint, HA stores a reference to the blueprint plus your input values. The actual automation logic stays in the blueprint file.

Supported domains: automation and script. Blueprints are NOT managed by HACS. They are imported via the HA UI or dropped into the directory manually. The community shares them on the HA forum's Blueprints Exchange section.

Key advantage over manual automations: when you update the local blueprint file (by re-importing or manually replacing it), all automations derived from it inherit the changes after an HA reload. The blueprint file itself does not auto-update from the source URL; that's a manual step.

The four ways to install Home Assistant (and why it matters)

There are different ways to put Home Assistant on a computer, and the method you pick determines what features you get. The simplest analogy:

Home Assistant OS = buying an iPhone. Everything works, all features available, Apple (the HA team) manages the system for you.

Home Assistant Container = buying an Android phone and installing your own apps. More control, but you manage more yourself. Some "app store" features (add-ons) aren't available.

The other two methods (Supervised and Core) are less common and not recommended for new setups.

If you're setting up fresh: go with Home Assistant OS. Flash it to a Raspberry Pi, a mini PC, or run it as a virtual machine. You get everything, including the add-on store.

Home Assistant OS (recommended)

This is the full package. You flash the HA operating system onto a device (Raspberry Pi, Intel NUC, old laptop, or a VM) and it manages everything: the system, updates, backups, add-ons. You get all features: integrations, add-ons, HACS, blueprints, one-click updates.

The downside: the device is dedicated to Home Assistant. You can't easily use it for other things.

Home Assistant Container

This is just the HA application running as a Docker container on a computer you manage yourself. Good if you already have a Linux server running other things. You get integrations, HACS, and blueprints, but not add-ons. If you need Mosquitto, Node-RED, or other add-on services, you install them yourself as separate Docker containers.

This is often the source of frustration: you follow a tutorial that says "install the Mosquitto add-on," but there is no add-on store in your setup. You have to translate those instructions into running a separate Mosquitto Docker container and connecting HA to it via the MQTT integration.

The other two (not recommended)

Supervised (you manage the OS, HA manages containers) and Core (bare Python install, no Docker) still exist but are not recommended for new setups. If you're starting fresh, go with HA OS or Container.

Bottom line: If a tutorial says "install this add-on" and you can't find the add-on store, your installation method is the reason. You're either on Container (no add-ons available) or something went wrong during setup. Check Settings → System → Repairs for clues.

The #1 user experience failure in the HA ecosystem: tutorials don't specify which installation method they assume. A user on Container follows a tutorial written for HA OS, hits a wall at "install the add-on," and has no idea why.

Home Assistant OS

A full Linux OS (a custom Linux distribution built with Buildroot) that boots directly into the HA Supervisor. Includes: HA Core (Python app), Supervisor (Docker orchestrator), and a managed OS layer that handles updates, snapshots, and network config. Runs on bare metal (Pi, NUC, x86), or as a VM image (VMDK, VHDX, OVA, QCOW2). The Supervisor manages Core as a container, add-ons as separate containers, and provides the ingress proxy.

Home Assistant Container

Just the ghcr.io/home-assistant/home-assistant Docker image. No Supervisor, no add-on support. Config directory is bind-mounted from the host. Users run companion services (Mosquitto, Z2M, Node-RED, MariaDB, Frigate, etc.) as standalone Docker containers, typically via docker-compose.yml. Communication happens over Docker networks and environment variables, connected to HA via integrations.

Why add-ons need the Supervisor

The Supervisor is a Docker container (ghcr.io/home-assistant/aarch64-hassio-supervisor) that orchestrates other containers via the Docker socket. It:

  • Pulls add-on images from registries
  • Manages container lifecycle (start, stop, restart, update)
  • Provides a bridge network (172.30.32.0/23) with DNS resolution between containers
  • Runs an ingress reverse proxy so add-on web UIs appear inside the HA frontend
  • Exposes the add-on store API

Without the Supervisor, none of this orchestration exists. There's no API to discover add-ons, no proxy for ingress, no container management. The add-on store UI in HA Core checks for the Supervisor API and simply doesn't render if it's absent.

Other methods (not recommended for new installs)

Supervised: Supervisor installed on user-managed Debian. Full add-on support but the HA team can't guarantee OS-level stability since they don't control the host. Strict requirements (Debian only, specific OS agent version). Recommended migration: backup → restore to HA OS.

Core: HA installed in a Python virtualenv. No Docker at all. No add-ons, no Supervisor. For advanced users who want maximum control. Migration path: backup → restore to HA Container or OS.

Why installing things on Home Assistant feels so hard

Imagine if your iPhone had three different App Stores, each with different apps, and some apps needed you to install a piece from one store and a piece from another, and one of the stores required you to sign up for a completely separate website first. That's Home Assistant right now.

It's not you. The system is genuinely confusing. The good news: once you know which "store" to look in for what, it gets much easier. And tools like Claude Code can handle the messy parts for you.

Here are the specific things that trip people up:

  • Three places to install things. Built-in integrations (Settings → Devices & Services), add-ons (Settings → Add-ons), and HACS (a separate panel). There's no unified search. You have to know which one you need.
  • Tutorials don't say which install method they assume. You follow a guide that says "install the Mosquitto add-on." You can't find it. Turns out it's because you're running HA Container, which doesn't have add-ons. The tutorial never mentioned that.
  • HACS needs a GitHub account. You just want a nicer-looking thermostat card. HACS makes you create a GitHub account, go to a separate website, enter a code, authorize an app. It feels like airport security for a dashboard widget.
  • Some things need both an add-on AND an integration. MQTT is the classic: you install the Mosquitto add-on (the broker), then separately add the MQTT integration (the connection). The tutorial might only mention one of those steps.
  • Restarts. After installing custom integrations via HACS, you have to restart all of Home Assistant. Forgot to restart? It looks like the install didn't work. You install it again. Now you have a mess.

None of these are your fault. They're real design problems in the HA ecosystem that the team is slowly improving.

The root cause is architectural: Home Assistant grew organically from a developer hobby project into the world's largest open-source home automation platform. Its extension mechanisms were designed at different times, by different teams, for different audiences:

  • Integrations (2013+) were built for developers adding device support to Core.
  • Add-ons (2017+) were built when the Supervisor was created to give HA OS users a way to run companion services.
  • HACS (2019+) was built by one community member who wanted an "app store" for custom integrations, since the official team had no plans to build one.
  • Blueprints (2020+) were added to lower the barrier for sharing automations.

Each one works well in isolation. The problem is discovery and onboarding: a new user doesn't know the taxonomy, and the UI doesn't guide them. The 2025 "Apps" rename and UI consolidation is a step toward fixing this, but it's going to take time.

Home Home Automation

Smart home setups always involve some YAML config file or a script that almost works. Claude Code can read those files, fix the errors, and write new automations for you.

Whether it's Home Assistant, Node-RED, Homebridge, or shell scripts that run on a schedule, you describe what you want and it builds it.

Things you can say to it

you: "Write a Home Assistant automation that turns off porch lights at sunrise"
you: "This YAML config has an error somewhere, can you find it?"
you: "Create a script that texts me when my garage door has been open for 10 minutes"
you: "Help me install HACS without me having to mess with GitHub tokens"
you: "Set up a Node-RED flow that logs my thermostat temperature every hour"
The key difference from ChatGPT: When you paste code into ChatGPT, it gives you a suggestion and you have to figure out where to put it. Claude Code actually reads your files, makes the changes, and runs them. It's not giving you advice. It's doing the work.
Getting Started

Set it up in five minutes

Open Terminal (hit Cmd + Space, type "Terminal," hit Enter). Then paste these commands one at a time. Each one finishes before you run the next.

1

Install Homebrew

This one-liner downloads and installs everything. It'll ask for your Mac password (the cursor won't move while you type it, that's normal).

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

When it finishes, it may print some lines saying "Add Homebrew to your PATH." If so, copy and paste those lines too. If it doesn't say that, you're good.

2

Install Node.js

Claude Code needs this to run. One command, takes about 30 seconds.

brew install node
3

Install Claude Code

This installs it globally so you can run it from anywhere.

npm install -g @anthropic-ai/claude-code
4

Launch it

Type claude and hit Enter. A browser window opens for sign-in on first run.

claude
Account options: Claude Code works with a free claude.ai account (limited usage), a Pro subscription ($20/mo for heavier use), or pay-as-you-go API credits at console.anthropic.com (add $5, use until it runs out). Start free and upgrade if you like it.
Quick Reference

Commands you'll actually use

Homebrew

# Install a GUI app
brew install --cask rectangle

# Install a command-line tool
brew install jq

# Update everything
brew upgrade

# Search for something
brew search postgres

# See what you've installed
brew list

# Remove something
brew uninstall something

Claude Code

# Start Claude Code
claude

# Start in a specific folder
cd ~/Documents && claude
